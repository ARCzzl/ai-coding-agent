## 二、面试常见问题与回答思路

### Q1: 你的项目做了什么？为什么选这个方向？

**回答要点**:
- 传统客服机器人只能做简单的 FAQ 匹配，无法理解复杂问题
- RAG（检索增强生成）可以让 LLM 基于真实知识库回答，而不是凭记忆"编造"
- 选择汽车领域是因为：数据可得（产品手册 PDF）、业务场景清晰（售前咨询）、用户需求明确

**话术模板**:
> 我做的项目是一个基于 RAG 技术的智能汽车销售顾问。选择 RAG 方向是因为它是当前大模型落地的核心范式——解决 LLM 的幻觉问题和知识时效性问题。我选了领克汽车的场景，把产品手册 PDF 作为知识库，构建了从文档解析到智能对话的完整链路。

---

### Q2: 你的 RAG 系统用了哪些检索策略？为什么？

**回答要点**: 这是核心考察点，要深入讲

**话术模板**:
> 我的检索 pipeline 分三层：
>
> **第一层-初检索**：混合检索（Hybrid Retrieval）
> - Dense检索（向量相似度）：负责语义匹配，比如"省油的车"能匹配到"油耗低"
> - Sparse检索（BM25+中文分词）：负责关键词精确匹配，比如"领克03"、"2.0T"
> - 加权融合：向量0.7 + BM25 0.3，语义为主、关键词兜底
>
> **第二层-重排序**：Cross-Encoder Reranker（BGE-reranker-large）
> - 初检索用 Bi-Encoder（分别编码 query 和 doc → 速度快）
> - 重排序用 Cross-Encoder（联合编码 query+doc → 精度高）
> - 这就是"召回→精排"的两阶段策略
>
> **第三层-查询优化**：Multi-Query Rewriting
> - 用户问题"03多少钱" → LLM 重写为"领克03价格"、"领克03落地价"、"领克03全款分期价格"
> - 多个等价查询分别检索 → 合并去重 → 大幅提高召回率
>
> 这个设计的核心思想是：**单一检索策略总有盲区，多层互补才能提高鲁棒性**。

---

### Q3: 为什么要用 RAGAS？不做评估行不行？

**回答要点**:
> 不做评估就是在"盲调"。我设计了5个测试问题、标注了参考答案，然后用 RAGAS 的四项指标（忠实度、答案相关性、上下文召回率、上下文精确度）来量化每次改动的影响。
>
> 比如：
> - 我把 chunk_size 从 300 调到 512，Recall@5 从 0.60 升到 0.72
> - 加了 Reranker 后，Faithfulness 从 0.78 升到 0.88
>
> 这些数字让优化有了方向，面试时可以说出"我做了什么、效果提升了多少"，这就比"我觉得效果好"强很多。

---

### Q4: 为什么用 Agent 框架而不直接调 LLM？

**回答要点**:
> Agent 的核心价值是**自主决策能力**。
>
> 比如用户问"分期买领克03首付30%，月供多少"，Agent 需要：
> 1. 判断这是计算类问题 → 调用 calculate_car_loan 工具
> 2. 参数是"领克03"、"30%"、"默认3年"
> 3. 如果车型模糊，先调用 car_info_rag 确认车型
>
> 如果不用 Agent，就需要写一堆 if-else 做意图识别和路由，维护成本极高。
> Agent = LLM + 工具 + 推理规划，让模型自己决定"什么时候该做什么"。

---

### Q5: 你遇到了什么技术挑战？怎么解决的？

**准备2-3个有深度的回答**:

**挑战1: 中文检索效果差**
> 最初直接用英文分词器，中文被切成单字。改用 jieba 分词+自定义 tokenizer 后，BM25 的效果提升明显。本质上理解了：不同语言的检索需要适配其语言特性。

**挑战2: 检索结果不精准**
> 初检返回5个文档给 LLM，有时关键信息不在前几条。加了 Cross-Encoder Reranker 后，Faithfulness 提升了约10个百分点。这教会我"召回和精排是两个不同的优化目标"。

**挑战3: 对话越长越"失忆"**
> 全量记忆导致 token 爆炸。调研后实现了三种策略（buffer/window/summary），不同场景切换。理解了"记忆管理本质上是上下文窗口的 token 预算管理"。

---

### Q6: LangChain 的内部原理你了解吗？

**考察框架理解深度，建议准备**:
> - AgentExecutor 的循环：Thought → Action → Observation → 重复直到 Final Answer
> - Chain 的本质是 Runnable 的 pipe 组合
> - Memory 的实现：往 prompt 里塞历史消息，本质是 prompt engineering
> - Tool Calling Agent 的原理：LLM 输出 function_call JSON → 框架解析 → 执行函数 → 结果注入 prompt
